# 去水印 go/no-go 评测 Run-book（operator 版）

> **一句话目的**：去水印引擎要不要做、做哪个，得用**你自有/授权的真实素材**在**你的笔电**上量一遍准确度和延迟才能定。这份文档带你一步步把那次测量跑出来。

## 为什么只有你能做这步

- **素材是你的**：评测要的是 owned/licensed 图（你有权处理的），不能用第三方/泄露站的内容。
- **延迟取决于你的机器**：去水印（inpaint）很吃 CPU，公开数据不可信——只有在你的目标笔电上量出来的「每张图几秒」才算数。
- **引擎没法替你选/打包**：候选引擎依赖重（torch/onnx），默认不进主程序；选哪个、装不装，是你看完数据后的决定。

代码侧已经就绪：评测脚手架（measurement-only，不碰生产代码）+ 画 mask 工具都在 `spikes/dewatermark/`，跑评测**不会**触发任何真实去水印动作。功能本身默认**锁死 + 引擎缺席**，合并/运行都安全。

## 第 0 步：环境

```bash
cd "<repo 根目录>"
./.venv/bin/python spikes/dewatermark/run_eval.py        # 先空跑一次合成基线，确认能跑
```
能看到一张表 + 末尾 `GO/NO-GO:` 就说明环境 OK（基线本来就该是 NO-GO，别担心）。

## 第 1 步：准备你的样本（按难度分桶）

每个 case 是一个子目录，放三张图：

```
你的样本目录/
  a/  thin logo / 平滑背景       ← 验收线 90%
    case1/clean.png             # 干净参考图（没有水印的版本）
         /watermarked.png       # 带水印的输入图
         /mask.png              # 要去掉的区域：L 模式，白(255)=去除，黑=保留
    case2/...
  b/  thin logo / 复杂纹理背景    ← 验收线 70%
  c/  半透明大面积 overlay        ← 验收线 90%
  d/  大面积/平铺/漂浮            ← v1 不达标也不卡（out-of-scope，仅记录）
  e/  压在人脸/主体上            ← 验收线 70%
```

- **`clean.png` 哪来？** 理想是「同一张图的无水印原件」（你自有素材通常拿得到）。拿不到无水印原件的，这桶就先少放或不放——没有 clean 参考，残留分数没法算。
- **每桶放几张？** 越多越可信；每桶 5～10 张就能看出趋势。
- 三张图缺一张，该 case 会被**自动跳过**（不会静默混入）。

---

## 第 2 步：给每张图画 mask

mask 标出「水印在哪、要 inpaint 哪块」。固定位置的平台水印通常就是一个矩形框：

```bash
# 用 --like 让 mask 自动对齐图片尺寸，--box 给水印框的左上(x0,y0)→右下(x1,y1)
./.venv/bin/python spikes/dewatermark/make_mask.py \
    --like 你的样本目录/a/case1/watermarked.png \
    --box 480 360 600 400 \
    --out 你的样本目录/a/case1/mask.png
```

- 水印有好几处？`--box` 可重复传多个。
- 框比水印**略大一圈**通常效果更好（给引擎一点上下文）。

## 第 3 步：跑评测

```bash
# 基线引擎（offline，纯 Pillow，故意弱）——先确认你的样本能被正确读入
./.venv/bin/python spikes/dewatermark/run_eval.py --samples 你的样本目录

# 接了真引擎后（见末尾），用 --engine 指定
./.venv/bin/python spikes/dewatermark/run_eval.py --samples 你的样本目录 --engine <引擎名>

# 想要机器可读结果就加 --json
```

## 第 4 步：读结果

表格每列：

| 列 | 含义 |
|---|---|
| `n` | 这桶有几张样本 |
| `PSNR` | 清理后 vs 干净参考的相似度（dB，越高越像） |
| `resid` | **mask 内的残留水印痕迹**（0～1，越低越干净）——最关键 |
| `pub%` | **可发布率**：这桶有几成达到「可发布」 |
| `bar` | 该桶的验收线 |
| `ms` | 每张图的**墙钟延迟**（毫秒）——你最该盯的「能不能忍」的数 |
| `verdict` | `PASS` / `BELOW BAR` / `out-of-scope` |

- **合成数据里**「可发布」= 残留 ≤ 阈值（机器近似）。**你的真实数据里**，「可发布」应由**人眼**判断——把 cleaned 结果逐张看过，决定能不能上。脚本只给你尺子，不替你下结论。
- 末尾 `GO/NO-GO`：所有**被卡的桶**（a/b/c/e）都 PASS → `GO`；任一不过 → `NO-GO/REVIEW`。桶 d 永远不卡。

---

## 第 5 步：BUILD or CUT 决策

- **GO**（被卡桶都过 + 延迟你能忍）→ **BUILD**：选定那个引擎，进入接线（见下）。
- **NO-GO** → 先分清是哪种：
  - **引擎不行**（真引擎也残留高）→ 换引擎再测，或 **CUT**。
  - **mask/样本问题**（框没盖准、clean 参考不对）→ 修样本重测。
  - **延迟不能忍**（准确度够但每张图太慢）→ 也是 NO-GO，CUT 或只在小批量场景用。
- **CUT 是合法且安全的选择**：功能保持默认锁死 + 引擎缺席，等于这部分能力不上线；要彻底干净，后续可移除 Batch 2 代码。**没有引擎 = DependencyError，没有具结 = 锁死**，所以即使不做决定，现状也安全。

---

## 接真引擎（给到时要 BUILD 的你/我）

脚手架按 `Engine` 协议插引擎，互相隔离：

- **MI-GAN-ONNX**：无 torch、约 29.5MB、可离线——首选候选。
- **static-ghost**：torch 系、仓库已有（视频向）——次选。

接法：在 `run_eval.py` 的 `ENGINES` 注册一个满足 `remove(watermarked, mask) -> Image` 的实现（真引擎在自己的隔离子进程/环境里跑，主程序绝不引入 torch/onnx），然后 `--engine <名字>` 评测。生产侧的隔离子进程 runner 已经建好（`src/lcp/adapters/media/dewatermark_runner.py`，已强制 `HF_HUB_OFFLINE/TRANSFORMERS_OFFLINE` 离线、禁止静默下载权重）。

## 安全红线（任何时候）

- **只对自有/授权素材去水印**；评测样本同理。
- 评测是**纯测量**，不碰生产、不上架。
- 真要启用：还要双人具结（提交人 ≠ 独立复核者）+ 授权依据，且引擎离线、权重固定——这些闸门代码里都在，默认全锁。
