# 内容流水线升级 — 需求文件（爬取 / 封面 / 水印 / AI 文案模板）

> 一句话定位：这是对现有 `lcp`（Local Content Processor）的**合规框内加厚**，不是转型。
> 四个功能里三个已有地基（爬取、封面拼接、受限改写 LLM），只有「去水印 / 加水印」是净新增。
> 两次方向题用户都选择**最小改动创始红线**：R16（受限改写防诽谤幻觉）维持不动；
> R2（不去水印掩盖来源）由本次开一个**有界例外**（仅自有/授权 + 独立复核），这属于**对 R2 的修订、需明示重新批准**，而非「红线没动」。
> 本文所有需求都在该前提下展开。基线为 [`2026-06-16-local-content-processor-requirements.md`](2026-06-16-local-content-processor-requirements.md)（R1–R40），本文用 `UR` 编号扩展，并标注与原 R 的关系。

## Problem Frame

经营者（非工程背景）运营一个吃瓜/爆料类内容站（分类：今日吃瓜 / 网红黑料 / 热门大瓜 / 海外吃瓜）。
现有 `lcp` 已能把「抓取 → 标准化 → 风控/查重 → 受限改写草稿 → 审核包」跑到「只差人按审核」的程度，
但对照其实际作业 SOP（《远程内容编辑发布流程 SOP_新媒体版》），**最耗时的几道手工活仍未自动化**：

- **文案撰写**：SOP 第七章一整套结构（标题、开头、一分钟快速看懂、事件经过、图说、FAQ、结尾）目前靠人逐篇手写。
- **封面制作**：1300×640 拼接已实现，但**打官方水印、安全区校验**仍靠人在 PS 里做。
- **水印处理**：SOP 第四/五章要求去三方水印 + 加官方水印，现有产品**完全没建**（且原本被红线排除）。
- **素材接收**：SOP「01 接收选题素材」是收一个混合素材包，现有本地导入体验偏薄。

痛点本质是：**合规与品质不能规模化地崩坏**的前提下，把这几道手工活降到「人只做审核」。
做不好的风险是真实人物爆料的**诽谤/幻觉/著作权**问题——这正是本文每条需求都要兜住的地方。

## 升级插入点（现有骨架不变，只在 4 个点加厚）

```
现有骨架（不变）                          本次升级插入的 4 个点
══════════════                          ════════════════════

--url / --dir
   │
   ▼
[Stage1 抓取 / 本地导入]  ───────────►  D 爬取(UR11-12)：限公开源 + 素材包导入；不碰 JS/登录墙
   │  raw_bundle
   ▼
[Stage2 处理]
   ├─ 媒体质检 / 标准化  ─────────────►  A 水印(UR1-4)：加自家水印 + 去水印(仅自有授权·声明闸门)
   │                                    C 封面(UR9-10)：拼接已有 + 打水印 + 安全区校验
   ├─ 风控 / 查重闸门（不变）
   ├─ LLM 受限改写       ─────────────►  B 文案(UR5-8)：提示词模板管理 + 图说/FAQ/小标题生成
   │                                       （叙事段仍绑定来源句，R16 不动）
   └─ linter / grounding（不变）
   │
   ▼
[Stage4 审核包] → 人工签核 → 人工上架  ─►  E 合规(UR13-16)：新动作全部只产待审包，无自动上架
```

> 注（评审补充）：B（UR5–UR8）是对 Stage2 现有 LLM 受限改写（R16）的**增强**，A/C 是插在媒体质检内的新步骤——三者都不新增独立 stage、不替代既有闸门。

## Requirements

**A. 水印处理 — 加自家水印 + 去水印（自有/授权限定）**
- UR1. **加自家水印**：对处理后的正文图与封面，按 config（logo 图档或文字、位置、透明度、留边）叠加本平台官方水印。
  水印为**品牌标识，不主张对底层素材的著作权**；来源仍保留在 audit / 来源栏，不掩盖来源（呼应原 R2）。
- UR2. **去水印预设锁死**，解锁需满足**权责分离 + 可核验依据**（评审 P0#1 决议）：
  (a) 填写**可核验授权依据**（合同号 / 授权 URL / 权属证明），非纯勾选；
  (b) 由**独立于该 job 提交/发布人的 reviewer**（取自白名单）复核并具名批准；
  (c) 声明内容、授权依据、批准 reviewer 写入 append-only audit。
  GUI 须**如实披露**：此声明是「具结非证明、记录责任非阻止侵权」（比照 R36/R37 诚实披露）。三者缺一 → 不去任何水印。
- UR3. 去水印**只处理已通过 UR2 授权复核的素材**；系统不自动判定第三方素材可否去水印，也不对未授权素材去水印。
  **诚实定性**（评审 P0#2）：原 R2「不得以**任何方式**掩盖权利人来源」是绝对禁令；本条是对 R2 开的**有界例外（= 修订）**，
  不是「收窄」——须作为对 R2 的明示修订重新批准，例外仅限自有/授权且经独立复核者。
- UR4. 去水印结果在 manifest 标记 `watermark_removed=true` + 授权依据 ref，成品仍走原媒体质检（模糊/尺寸）与人工校阅；
  inpainting 失败或低置信 → status=needs_revision，**不静默输出残缺图**。

**B. AI 文案 — 受限改写之上的提示词模板管理**
- UR5. 维持原 R16「受限改写」红线：叙事段（事件经过、FAQ 事实部分）绑定来源句、不自由杜撰，机器产出一律标 needs_human_review。
- UR6. 新增**提示词模板管理**：操作者可按栏目（今日吃瓜 / 网红黑料 / 海外吃瓜 / 热门大瓜…）维护可复用的提示词模板与文案骨架
  （标题结构、开头、一分钟快速看懂、小标题风格、FAQ 风格、结尾），可版本化、可在 GUI 选择套用。
- UR7. AI 负责生成**低风险结构件**：图片说明（caption）、FAQ 问答、分组小标题、标题候选；
  这些仍标待审、过 linter（原 R23），并满足 SOP 排版去重要求（小标题/FAQ/段落不得机械复用同一套）。
- UR8. 模板填充遵守原 R17 客观规则（标题 25–35 字、tags 3–5 个客观词、禁用夸大词、分类取自 config）；
  提示词模板内**不得内嵌诱导自由杜撰或越过 grounding 的指令**（呼应原 R35 注入防护，模板本身也视为受检对象）。
  - **评审补充（安全 P1#8）**：UR6 用户可编辑模板是**首次**把用户字符串引入可信 system prompt 区（现状 prompt 是硬编码常量）——
    须对模板做注入/越狱 lint，且模板**不能改写**「零能力 / 数据非指令 / grounding」这几条系统约束。
  - **评审补充（审核量 P0#4）**：B 组净效果必须是**每篇人工审核量下降**（见 Success Criteria 可量测项）；
    若图说/FAQ 的人工校对抵消了正文省下的时间，该批次视为未达标，须收缩 AI 生成范围。

**C. 封面 — 水印与安全区（拼接已有）**
- UR9. 复用现有 1300×640 拼接（1–4 图版式已实现）；新增**安全区校验**：主体须落在安全警戒线内，
  命中边缘/拥挤/头重脚轻启发式 → GUI 警告或 needs_revision（呼应 SOP 第六章与原 R13 的 P2 项）。
- UR10. 封面叠加官方水印（复用 UR1 加水印基元）；封面**不得出现文字、第三方水印、外链、黑边白边**
  （SOP 第六章硬规则）→ 由 linter / 媒体质检校验，命中即标 needs_revision。

**D. 爬取与素材导入（小幅，不碰高风险区）**
- UR11. 维持现有 Scrapy 公开源抓取 + 本地导入为**唯一通道**：不新增 JS 重页（Playwright 留到确有公开可合法源时再立项）、
  不绕登录/付费墙/反爬（原 R7/R8 维持）。范围限「公开网页 / RSS / 自有站 + 素材包本地导入」，
  明确**不抓微博/小红书/抖音/知乎**这类登录墙平台。
- UR12. 升级**素材包本地导入体验**：支持一个文件夹混合图/视频/文字记录的导入与归档（呼应 SOP「01 接收选题素材」），
  做完整性核对（可开启/播放、缺失项标注），复用现有 media_checker。

**E. 跨功能 — 合规闸门与 audit（继承原红线）**
- UR13. 所有新增动作（去水印声明、加水印、模板套用、安全区判定）逐步写入 per-job append-only audit；
  机密遮罩与 PII 治理沿用原 R19/R38/R39。
- UR14. 所有新增 stage **idempotent、dry-run 安全、不覆盖输入**（原 R29/R32）：dry-run 下不调 LLM、不写水印成品。
- UR15. 新增能力**不引入任何自动上架路径**（原 R26 与 Scope 维持）：去水印/加水印/AI 文案/封面都只产出待审包，
  人工审核 + 人工上架流程不变。

**F. 操作界面（GUI / CLI 1:1）**
- UR16. GUI/CLI 对等暴露新动作：去水印声明开关、加水印开关、栏目模板选择、安全区警告、封面预览；
  非技术操作者不写程式即可用（原 R33）；reviewer 身分与签核归属沿用原 R25/R37。

## 建议交付顺序（用户授权由本文排定）

按「日常省时 × 风险 × 代码依赖」排，理由附后：

- **批次 1（先做：日常最痛 + 低风险 + 解锁共享基元）**
  - 先建 **UR1 加水印基元**（一次建成，封面与正文图复用）→ 接 **UR9/UR10 封面水印 + 安全区**。
  - **并行** **UR5–UR8 AI 文案模板管理**（扩展现有 assembler，独立于媒体，省时最多）。
  - *理由*：SOP 里文案与封面是最耗时的手工活；封面拼接已有、文案 LLM 已有，边际成本最低；加水印是 UR9/UR10 的前置基元，先建一次。

- **批次 2（缺口最大但最重）**
  - **UR2–UR4 去水印**：需引入本地 inpainting 模型 + 「自有/授权」声明闸门 + 准确度 spike。
  - *理由*：唯一净新增、carrying cost 最高、使用频率偏低（occasional），且需准确度验证后才敢上。
  - **go/no-go（评审补充）**：spike 须先过验收线（可去水印类型覆盖 + 残留率 + 失败回退）；若不过、或实际使用量仍为 occasional，**批次 2 直接砍掉**，授权的偶发去水印退回人工 PS——它不服务文案/封面两大时间黑洞，砍掉不影响主目标。spike 决定的是「**要不要建**」，不只是「怎么建」。

- **批次 3（小幅，按需）**
  - **UR11–UR12 爬取/导入增强**。*理由*：现有 Scrapy + 本地导入已可用，缺口最小。

- **横切**：**UR13–UR16**（audit / dry-run / 无自动上架 / GUI）随各批次同步落地，不单列。

## Success Criteria

- 非技术操作者能在 GUI 里：选栏目模板 → 跑出含**图说/FAQ/小标题**的待审草稿（叙事段仍绑来源句、标待审），文案手工量显著下降。
- **可量测（评审补充）**：操作者每篇的「审核/修改触点数」或人工耗时，相对升级前基线（可从现有 `lcp.db` 每阶段计时提取）下降到设定目标；**若新增 AI 产物反而抬高每篇审核量，视为未达标**（防止「自动化反而增加人工」）。
- 封面能**自动拼接 + 打官方水印 + 安全区警告**，命中「文字/三方水印/外链/黑白边」时能拦下（SOP 第六章硬规则）。
- 加水印对所有处理后图片/封面生效；**去水印仅在「可核验授权依据 + 独立 reviewer 复核」后**才动，且全程 audit 可溯、来源不掩盖。
- AI 文案**未松绑 R16**：无自由生成全文路径；模板本身受注入检查；机器产出全标待审。
- **诽谤/未证实可量测**（评审 P0#3）：R4/R5 对「诽谤性陈述 / 未证实当事实」的检出，须用一小组标注样本给出可接受的**漏检率**验收线——规模化前必须有数字，否则「合规优先」不可证伪。
- 爬取范围**诚实收窄**：只走公开源 + 本地导入，GUI/文档明示不抓登录墙平台、不绕反爬。
- 新增能力**零自动上架**：dry-run 不动外部系统、不写成品；每个新动作都有 audit。
- **可量测**：去水印的「可去类型 / 残留率 / 失败回退」、安全区判定的误判率，需有一小组标注样本验收（沿用原 spike 思路）。

## Scope Boundaries（本次不做）

- **不做**对第三方/未授权素材的去水印掩盖（仅限自有/授权 + 独立复核闸门）。
- **不做**自由生成全文（维持受限改写；AI 只生成低风险结构件）。
- **不做**微博/小红书/抖音/知乎等登录墙平台抓取，**不做**登录/付费墙/反爬绕过，**不做** Playwright JS 重页（延后）。
- **不做**自动建后台草稿 / 自动发布（沿用 MVP Stage 5/6 边界）。
- **不做**封面 AI 生图（只拼接现有素材 + 水印 + 安全区）；如需 AI 生图另立项。

## Key Decisions

- **水印 = 自有/授权限定 + 独立复核闸门（修订 R2）**：满足 SOP 第四/五章。**诚实定性**：这是对 R2 绝对禁令开的**有界例外**，
  靠「可核验授权依据 + 权责分离复核 + 诚实披露」把风险压到可辩护，而非「R2 没动」。
- **AI = 受限改写 + 模板管理**：原 R16 维持。**诚实定性**（评审 P0#3）：受限改写只压住**幻觉/杜撰**风险，
  **压不住「转述来源诽谤」风险**——忠实绑定一句诽谤性/未证实的爆料原文，等于忠实转述诽谤（本身可诉）。
  真正的防线在**上游 R4/R5 选源闸门**（诽谤性陈述检测 + 未证实语气），规模化恰恰给它加压；故须给 R4/R5 检出补一条可量测验收线（见 Success Criteria）。
- **爬取 = 只公开源 + 本地导入**：避开 JS/登录墙平台的法律、ToS 与反爬风险。理由：carrying cost 与风险最低。
- **加水印为共享基元**：封面与正文图复用同一实现，先建一次。
- **优先级 = 文案 + 封面先行、去水印压后**：按日常省时 × 风险 × 依赖排序。

## Dependencies / Assumptions

- 去水印需引入**本地 inpainting 模型**（如 LaMa 类），**素材不外传**（呼应隐私/PII 治理）；准确度需 spike 验证。
- 加水印需提供**官方水印资产**（透明 PNG / 字体）与位置规范。
- 复用既有 Pillow（图像）、ffmpeg/ffprobe（视频）、OpenAI-compatible LLM client（keyring + base_url）。
- 提示词模板管理沿用现有 config + keyring 机制，无需新外部服务。
- 封面拼接（`normalizer.py` 的 1–4 图版式）与 media_checker 闸门已存在，作为 UR9/UR10/UR12 的基座。
- **（评审补充）planning 前须把《远程内容编辑发布流程 SOP_新媒体版》第四/五/六/七章 commit 进 `docs/spec/`**：UR2–UR4 / UR9 / UR10 / UR12 多处以 SOP 为据，但 SOP 目前不在 repo，实作者无档可依（基线 doc 已提过同一要求，至今未做）。
